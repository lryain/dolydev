# modules/doly 模块介绍

## 这个模块是做什么的

`modules/doly` 是多利机器人的主控守护进程，也是整个系统的“大脑”。它统一接收来自语音、传感器、定时器、视觉、交互等各类事件，并决定机器人下一步应该做什么。

## 小白可以怎么理解它

如果机器人里只能选一个最核心的模块，那通常就是这个模块。

它不像底层驱动那样直接操作硬件，也不像动画引擎那样只负责执行动作，而是负责：

- 当前状态是什么
- 收到一个事件后要不要响应
- 响应时应该调用哪个模块
- 多个能力之间怎么协同

## 它主要负责什么

- 管理状态机，例如 `IDLE`、`ACTIVATED`、`EXPLORING`、`SLEEPING`
- 统一事件总线，接收和分发系统事件
- 处理语音命令和命令映射
- 协调动画系统、眼睛引擎、Widget、TTS、FaceReco、ToF 等模块
- 承载定时任务、交互逻辑、探索逻辑和行为管理逻辑

从 `daemon.py` 可以直接看出它初始化了大量管理器和客户端，例如：

- `WidgetManager`
- `VoiceCommandManager`
- `AnimationManager`
- `FaceRecoManager`
- `TTSManager`
- `VisionModeManager`
- `EyeEngineClient`
- `WidgetServiceClient`

## 输入与输出

### 输入

- 语音命令
- 串口唤醒/离线命令
- ToF、FaceReco、Widget、眼睛状态等系统事件
- 定时器、状态机和任务引擎产生的内部事件

### 输出

- 调用动画系统播放行为
- 调用眼睛引擎切换视觉表现
- 调用 Widget 显示时钟/定时器
- 调用 TTS、音频、视觉模式、探索模式等能力

## 它依赖谁、又被谁依赖

### 它依赖谁

- `modules/animation_system`
- `modules/eyeEngine`
- `libs/widgets`
- `libs/FaceReco`
- `libs/tof`
- `libs/tts/edge-tts`
- 以及各类客户端、事件总线和配置文件

### 谁依赖它

- 作为系统主控，通常没有更高层的业务模块依赖它
- 它本身就是多利运行时的调度中心

## 一个典型调用链

1. 串口或语音系统产生一个用户命令
2. `modules/doly` 通过事件总线接收到事件
3. 状态机和命令管理器判断当前是否该执行
4. 决定调用动画、眼睛、TTS、Widget、视觉或探索模块
5. 机器人完成一次交互或动作反馈

## 什么时候你应该看这个模块

- 机器人整体行为不协调
- 命令明明识别到了，但没有触发动作
- 一个模块单测正常，但在整机中没有按预期工作
- 你要增加新的跨模块功能

## 建议先看哪些文件

- `README.md`
- `daemon.py`
- `state_machine.py`
- `event_bus.py`
- `managers/`
- `clients/`

## 一句话总结

`modules/doly` 不是单一能力模块，而是把所有能力串起来、让机器人“像一个整体工作”的地方。
