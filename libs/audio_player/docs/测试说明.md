# audio_player 测试说明

本文档用于说明 `libs/audio_player/testing` 目录下各个测试脚本的用途、运行前提、执行步骤和结果判定方法。测试目标是验证 `audio_player_service` 的命令接入、状态发布、重复播放、优先级控制、抢占、ducking 和停止控制是否正常。

## 1. 测试范围

当前测试目录中的脚本主要覆盖以下能力：

- 通过 ZeroMQ 向 `audio_player_service` 发送播放/停止命令。
- 验证 `cmd.audio.play` 的单次播放、重复次数播放和按时长循环播放。
- 验证 `cmd.audio.stop` 对 `alias`、`uri`、`all` 等目标的停止效果。
- 监听 `status.audio.playback` 状态流，检查服务是否持续发布播放状态。
- 在组合测试中检查优先级、抢占、ducking 和并发上限等行为。

## 2. 测试前置条件

### 2.1 运行环境

- 需要能运行 `audio_player_service`。
- 需要 Python 3。
- 需要安装 `pyzmq`。

如果当前 Python 环境缺少 `pyzmq`，建议先激活仓库常用虚拟环境：

```bash
source /home/pi/dolydev/.venv/bin/activate
```

### 2.2 服务端要求

测试脚本默认连接以下 IPC socket：

- 命令端点：`ipc:///tmp/doly_audio_player_cmd.sock`
- 状态端点：`ipc:///tmp/doly_audio_player_status.sock`

因此必须先启动 `audio_player_service`。如果服务未运行，脚本通常会报连接失败、超时或没有状态输出。

### 2.3 启动方式

推荐在调试阶段以前台方式启动服务，便于直接查看日志：

```bash
cd /home/pi/dolydev/libs/audio_player/build
./audio_player_service
```

如果已经通过 systemd 安装，也可以用服务方式启动：

```bash
sudo systemctl start doly-audio-player.service
sudo systemctl status doly-audio-player.service
```

## 3. 测试脚本说明

### 3.1 `test_repeat_playback.py`

路径：`libs/audio_player/testing/test_repeat_playback.py`

用途：验证 `cmd.audio.play` 的重复播放能力，重点覆盖：

- `playback_mode: repeat_count`
- `playback_mode: repeat_duration`
- `repeat_interval_ms`
- 服务端重复播放日志

脚本行为：

1. 连接 `ipc:///tmp/doly_audio_player_cmd.sock`。
2. 发送一条 `cmd.audio.play` 命令。
3. 分别测试重复次数模式和按时长模式。
4. 输出服务端回复。
5. 提示用户检查服务日志中的重复播放信息。

典型输出包含：

```text
✅ Connected to audio_player service
📤 Sending: { ... }
✅ Reply: { ... }
```

服务端日志中应能看到类似内容：

```text
[AudioPlayer] repeat: alias=sfx_camera_click count=1/3 interval_ms=500
[AudioPlayer] repeat_duration restart: ...
[AudioPlayer] repeat_duration complete: ...
```

#### 运行方式

```bash
cd /home/pi/dolydev/libs/audio_player/testing
python3 test_repeat_playback.py
```

#### 判定标准

- REQ/REP 请求能收到回复。
- `repeat_count` 模式按设定次数重复播放。
- `repeat_duration` 模式能在设定时长内持续循环。
- 服务日志里能看到重复播放相关记录。

### 3.2 `test_multi_audio.py`

路径：`libs/audio_player/testing/test_multi_audio.py`

用途：验证多种播放场景的组合行为，是一个偏手工的集成测试脚本，主要关注：

- 优先级控制。
- `preempt` 抢占。
- `ducking` 降音量。
- 并发播放上限。
- 重复播放模式。

脚本特点：

- 通过一个 `paths` 列表逐条发送测试命令。
- 大量示例条目被注释，便于按需打开测试。
- 既支持 `alias`，也支持 `uri`。

建议使用方式：

1. 先保留少量关键条目，例如背景音乐、短音效、重复播放条目。
2. 逐步打开更多条目，观察服务端响应和实际播放效果。
3. 重点观察是否出现优先级覆盖、抢占生效、ducking 生效和并发限制生效。

#### 运行方式

```bash
cd /home/pi/dolydev/libs/audio_player/testing
python3 test_multi_audio.py
```

#### 适合验证的场景

- 背景音乐播放时插入短音效。
- 高优先级声音抢占低优先级声音。
- 带 `ducking: true` 的声音是否能压低其他声音。
- `repeat_count` 和 `repeat_duration` 命令是否与普通播放兼容。

### 3.3 `test_audio_stop.py`

路径：`libs/audio_player/testing/test_audio_stop.py`

用途：验证停止命令的处理逻辑，重点观察：

- `cmd.audio.stop` 是否能停止指定目标。
- `alias: all` 是否能停止所有播放中的音频。
- 未来如果扩展到 `uri` 停止，也能作为验证脚本使用。

脚本行为：

- 连接 `ipc:///tmp/doly_audio_player_cmd.sock`。
- 按 `paths` 列表逐项发送停止请求。
- 每次发送后等待服务回复。
- 默认示例中使用 `alias: all`。

#### 运行方式

```bash
cd /home/pi/dolydev/libs/audio_player/testing
python3 test_audio_stop.py
```

#### 使用建议

- 先在另一个终端中播放一段音频，再运行停止测试。
- 如果要测试按单个 `alias` 停止，可以把 `paths` 改成具体别名。
- 如果要测试按 `uri` 停止，可以在脚本里取消对应的注释项。

#### 判定标准

- 服务端返回正常回复。
- 正在播放的音频被正确终止。
- `all` 停止后所有活跃声音都消失。

### 3.4 `check_status.py`

路径：`libs/audio_player/testing/check_status.py`

用途：监听 `status.audio.playback` 状态发布，查看服务当前是否有播放任务、活跃声音列表是否正确变化。

脚本行为：

1. 连接 `ipc:///tmp/doly_audio_player_status.sock`。
2. 订阅主题 `status.audio.playback`。
3. 在约 5 秒内持续打印收到的状态消息。
4. 最后输出最近日志检查提示。

#### 运行方式

```bash
cd /home/pi/dolydev/libs/audio_player/testing
python3 check_status.py
```

#### 判定标准

- 能持续收到状态消息。
- 状态中的 `running`、`active_sounds`、`sounds` 等字段随播放情况变化。
- 播放开始和结束时状态有明显变化。

## 4. 推荐测试流程

建议按以下顺序执行，最容易定位问题：

1. 启动 `audio_player_service`。
2. 运行 `check_status.py`，确认状态发布正常。
3. 运行 `test_repeat_playback.py`，确认重复播放功能正常。
4. 运行 `test_multi_audio.py`，确认组合播放、优先级和 ducking 正常。
5. 运行 `test_audio_stop.py`，确认停止命令有效。

如果只想快速验证重复播放，可以只执行第 1 步和第 3 步。

## 5. 结果观察要点

### 5.1 命令回复

测试脚本通过 REQ/REP 方式和服务通信，因此每次发送命令后都应收到回复。常见回复应至少包含：

- `ok: true`
- `alias`
- `path`
- `priority`

如果命令参数无效，服务可能返回 `ok: false` 和错误信息。

### 5.2 日志观察

建议重点查看 `audio_player_service` 日志，尤其是以下信息：

- 是否出现 `repeat:` 相关日志。
- `repeat_count` 是否按预期递增。
- `repeat_duration` 是否能在时长到达后结束。
- `cmd.audio.stop` 是否能及时中止播放。
- `ducking` 和 `preempt` 是否按预期触发。

### 5.3 状态观察

状态订阅脚本应能看到播放中的音频条目变化。测试时可重点确认：

- 有播放时状态不是空的。
- 停止后活跃列表清空。
- 组合播放时活跃条目数不超过配置的并发上限。

## 6. 常见问题与排查

### 6.1 连接被拒绝

现象：脚本报 `Connection refused`、`No such file or directory` 或一直超时。

排查：

```bash
sudo systemctl status doly-audio-player.service
```

或者确认前台服务是否已经启动，并检查 `/tmp/doly_audio_player_cmd.sock` 是否存在。

### 6.2 脚本没有输出状态

现象：`check_status.py` 没有打印任何状态。

排查：

- 确认服务正在运行。
- 确认订阅的是 `ipc:///tmp/doly_audio_player_status.sock`。
- 确认主题是 `status.audio.playback`。

### 6.3 没有声音

现象：命令有回复，但没有实际播出声音。

排查：

- 检查音频设备配置。
- 确认当前音频文件或 `alias` 对应的资源存在。
- 查看服务端日志是否有加载失败、设备打开失败或播放失败信息。

### 6.4 重复播放没有按预期执行

现象：`repeat_count` 或 `repeat_duration` 看起来只播放了一次。

排查：

- 确认命令中设置了 `playback_mode`。
- `repeat_count` 时确认 `play_count > 0`。
- `repeat_duration` 时确认 `play_duration_ms > 0`。
- 查看服务日志是否打印了重复播放调试信息。

### 6.5 停止测试没有效果

现象：发送停止命令后，声音仍然在播放。

排查：

- 确认停止目标写对了，尤其是 `alias` 是否和正在播放的资源一致。
- 先用 `check_status.py` 确认当前活跃声音列表。
- 如果测试的是 `all`，确认脚本里的请求确实发出了 `alias: all`。
