# libs/audio_player 模块介绍

## 这个模块是做什么的

`libs/audio_player` 是多利机器人的音频播放引擎。它基于 `miniaudio` 构建，负责把音效、提示音、语音合成结果等音频内容稳定地播放出来。

它的定位不是“只放一个 wav 文件”，而是作为机器人统一的音频播放服务，供其他模块通过命令方式调用。

## 小白可以怎么理解它

当机器人要说话、播放提示音、播放闹钟或执行带声音的动画时，真正负责“把声音从喇叭里放出来”的通常就是这个模块。

你可以把它理解成多利系统里的“音频输出中心”。

## 它主要负责什么

- 接收音频播放命令，例如 `cmd.audio.play`
- 按优先级、时长、重复次数等规则控制播放
- 为闹钟、提示音、动画音效、TTS 结果提供统一播放能力
- 承担实际音频设备输出

从现有文档可以确认，它已经支持围绕 `cmd.audio.play` 的扩展设计，例如：

- 单次播放
- 按次数重复播放
- 按持续时长循环播放
- 优先级和抢占控制

## 输入与输出

### 输入

- 音频播放命令
- 音频别名、文件路径、优先级、音量、重复策略等参数
- 来自 TTS 或其他模块的待播放音频文件

### 输出

- 实际的扬声器播放结果
- 播放过程中的状态变化和响应结果

## 它依赖谁、又被谁依赖

### 它依赖谁

- `miniaudio`
- 本地音频资源文件和配置

### 谁会调用它

- `modules/doly`：主控模块会触发各种播放请求
- `modules/animation_system`：动画中的声音块最终要落到音频播放能力
- `libs/tts/edge-tts`：语音合成完成后，通常需要交给音频播放模块播出
- `libs/widgets`：时钟/定时器类提示音也会用到它

## 一个典型调用链

1. 某个上层模块决定要播放声音
2. 它发送音频播放命令和参数
3. `audio_player` 解析命令并加载音频资源
4. 播放器按优先级和播放策略输出声音

## 什么时候你应该看这个模块

- 机器人“不说话”但 TTS 文件已经生成了
- 某些音效无法播放或被意外打断
- 闹钟、提示音、重复播放逻辑异常
- 你要增加新的音频控制协议字段
