# modules/animation_system 模块介绍

## 这个模块是做什么的

`modules/animation_system` 是多利机器人的动画编排引擎。它负责读取 Blockly 导出的 XML 动画文件，把动画里的眼睛、灯光、声音、机械臂、底盘动作解析成可执行的步骤，然后按照时序和并发规则执行。

## 小白可以怎么理解它

这个模块不是直接控制硬件，而是把“一个完整动作”拆成很多小动作并按顺序执行。

例如“开心一下”这个动作，内部可能包含：

- 眼睛播放一个高兴表情
- LED 变色
- 播放一段音效
- 手臂抬起
- 底盘稍微移动一下

这些动作如何组合、同步、延时，都是 `animation_system` 负责。

## 它主要负责什么

- 解析 XML 动画文件
- 根据动画块类型构建执行计划
- 支持顺序执行和并发执行
- 将高层动画块映射到具体硬件接口
- 提供动画缓存、暂停、恢复、停止等能力

`README.md` 中明确给出了它的三个核心组件：

- `AnimationManager`：加载和管理动画资源
- `AnimationExecutor`：真正执行动画序列
- `HardwareInterfaces`：抽象眼睛、LED、声音、机械臂、底盘等硬件接口

## 输入与输出

### 输入

- XML 动画文件
- 动画分类、等级、动画名称
- 来自上层主控或业务模块的播放请求

### 输出

- 对眼睛、LED、声音、手臂、底盘等接口的调用
- 动画运行状态，例如正在执行、暂停、停止

## 它依赖谁、又被谁依赖

### 它依赖谁

- `modules/eyeEngine` 或其它眼睛实现
- 音频、LED、机械臂、底盘等具体控制接口
- `SDK` 或其它硬件适配层提供的能力

### 谁会用它

- `modules/doly`：主控模块通常通过它触发行为动画
- 测试脚本、命令行工具、行为管理器也会直接调用它

## 一个典型调用链

1. 上层决定播放某个行为动画
2. `AnimationManager` 找到对应 XML 文件
3. 解析器把 XML 转成动画块序列
4. `AnimationExecutor` 根据块类型调用对应硬件接口
5. 机器人表现出一整套动作

## 什么时候你应该看这个模块

- 动画文件能找到，但执行效果不对
- 多个动作的先后顺序或并发逻辑有问题
- 你要新增一种 Blockly 动画块
- 你要把一个新的硬件能力接入动画系统
