# libs/drive 模块介绍

## 这个模块是做什么的

`libs/drive` 是多利项目当前的底层驱动模块。它把机械臂、底盘、LCD、LED、舵机、传感器、ToF 使能和扩展 IO 等能力封装成统一的 C++ 驱动层，并向上提供服务进程和可复用库接口。

如果说 `modules/doly` 负责“想做什么”，那么 `libs/drive` 就负责“真正把硬件做出来”。

## 小白可以怎么理解它

它更像机器人里的“硬件操作系统”或“设备驱动层”：

- 上层告诉它要移动、点亮、使能、读取传感器
- 它负责把这些指令翻译成 GPIO、I2C、串口、ZMQ 和具体硬件动作

和业务逻辑比起来，这一层更靠近设备，也更容易因为接线、地址、供电、时序问题出错。

## 它主要负责什么

- 舵机控制：角度设置、平滑运动、多舵机同步、摆动动作、自动保持/释放
- 扩展 IO：PCA9535 相关的引脚控制、触摸/悬崖识别、原始状态上报
- ToF 相关初始化：启动时做传感器地址配置和基础准备
- 电机与编码器：底盘相关控制、速度闭环、编码器读取、安全监测
- LED / LCD / 传感器相关底层封装
- 服务化运行：提供 `drive_service` 进程，通过 ZMQ 等方式接收控制并发布状态

从源码结构可以看出它的核心入口通常分布在：

- `src/drive_service.cpp`
- `src/drive_service_main.cpp`
- `src/servo_controller.cpp`
- `src/pca9535_service.cpp`
- `src/motor_controller.cpp`

## 输入与输出

### 输入

- 上层主控下发的控制命令
- SDK 或测试工具发起的驱动请求
- 配置文件、启动参数、传感器地址配置
- 硬件中断、GPIO 状态变化、编码器反馈、ToF 数据

### 输出

- 舵机、底盘、LED、LCD、传感器的实际硬件动作
- 运行状态、事件、传感器数据、错误信息
- 可供上层订阅的 ZMQ 消息或状态回调

## 它依赖谁、又被谁依赖

### 它依赖谁

- `libs/Doly/include` 中的官方硬件 API 头文件
- `libgpiod`
- `libzmq`
- `yaml-cpp`
- `Python3` 运行环境（部分桥接和工具链使用）

### 谁会用它

- `modules/doly`：把业务事件转成硬件动作
- `modules/eyeEngine`：眼睛/LCD 相关显示
- `libs/widgets`：时钟、定时器显示与 LCD 协调
- `SDK`：对外示例和开发接口
- `tests/` 和 `scripts/`：驱动验证、部署、维护

## 什么时候你应该看这个模块

- 机械臂、底盘、LCD、LED、舵机没有响应
- ToF 初始化或地址配置失败
- 传感器中断、状态上报、扩展 IO 事件不正常
- 底盘动作、舵机动作、驱动服务启动异常

## 建议先看哪些文件

- `README.md`：快速了解模块定位
- `src/drive_service.cpp`：服务主逻辑
- `src/servo_controller.cpp`：舵机控制能力
- `src/pca9535_service.cpp`：扩展 IO 和传感器相关能力
- `include/doly/`：公开接口
- `scripts/manage_service.sh`：部署和运行

## 一句话总结

`libs/drive` 是多利的当前底层驱动层，负责把“上层想做的动作”变成“硬件真的动起来”。
