# FaceReco 深度评估与建议

> 更新日期: 2026-04-12
> 结论: 经过本轮修复与功能测试，FaceReco 的 ZMQ 主链路已经可运行、可测试、可回归，但仍有少量工程化改进空间。

---

## 1. 模块现状概览

FaceReco 当前已经具备以下核心能力:

- 运行模式切换: IDLE、DETECT_ONLY、DETECT_TRACK、FULL、STREAM_ONLY、CUSTOM
- ZMQ 控制: 模式切换、拍照、录像开始/结束、人脸注册、更新、删除、查询
- 视觉能力: 人脸检测、跟踪、识别、陌生人发现、活体判断
- 数据管理: face_db.json 元数据持久化、descriptors.yml 描述子缓存、运行时描述子热更新
- 自动化测试: 可直接执行的 Python ZMQ 功能测试程序，带明确 PASS/FAIL 日志

这意味着模块已经从“设计文档存在、实现零散可用”推进到“可由外部进程通过 ZMQ 稳定控制，并能进行端到端回归验证”的阶段。

---

## 2. 本轮确认的关键问题

### 2.1 已修复问题

| 问题 | 表现 | 处理结果 |
|------|------|------|
| 旧工程路径硬编码 | 配置、构建和运行默认路径仍指向历史目录 | 已改为基于当前工程路径和配置解析 |
| 新人脸注册依赖 stdin | 服务运行中会阻塞等待终端输入 | 已改为发布 event.vision.face.new，由 ZMQ 命令驱动注册 |
| event.vision.face.new 缺失 | 文档承诺存在，实际不发事件 | 已补齐事件发布 |
| register 仅写数据库不更新识别库 | 注册成功后无法立即识别 | 已接入运行时 descriptor store 热更新 |
| STREAM_ONLY 和抓拍录像活跃态判断不稳 | 模式切换、抓拍、录像处理可能被主循环忽略 | 已补 active wait 和 capture request 唤醒逻辑 |
| 文件视频源 EOF 处理错误 | 文件播完后不重开 source，录像 stop 也可能卡住 | 已修复 ParallelVideoCapture 的 running 状态回收 |
| ready 事件易丢失 | 单次 PUB 广播可能因慢连接被订阅端错过 | 服务侧增加多次 ready 广播，测试侧增加 status fallback |
| 测试工作目录未清理 | face_db、descriptor cache、样本图可能污染下一次执行 | 测试前改为清理隔离目录 |
| 描述子 zScore 可能产生 NaN | 注册成功但识别始终返回 empty | 已修复数值稳定性，并在相似度比较时忽略非法分数 |

### 2.2 已验证通过的功能

本轮实际执行确认通过的强校验项:

- 服务启动与就绪判定
- DETECT_ONLY 模式下的稳定跟踪快照
- FULL 模式切换
- ZMQ 拍照及文件保存校验
- ZMQ 录像开始/停止及文件保存校验
- STREAM_ONLY 模式下抑制人脸检测事件
- 陌生人事件 event.vision.face.new
- 当前跟踪人脸注册 cmd.vision.face.register
- query.vision.face.list 查询校验
- 注册后识别命中新标签
- 更新姓名后再次识别命中新标签
- 删除后 query 不再返回被删记录

最近一次完整执行结果: all functional tests passed。

## 3. 设计与实现之间的偏差

### 3.1 数据源仍然存在“双源真相”问题

当前识别主路径仍然以图片数据集和描述子缓存为准，而 query.vision.face.list 以 face_db.json 元数据为准。

本轮为保证测试一致性，测试夹具会同时预置:

- 数据集图片 known_subject_0.jpg
- 对应的 face_db.json 元数据记录

这保证了“识别能命中”和“查询能返回”在测试场景中一致，但从工程设计上看，系统长期仍应收敛到单一事实来源。

建议:

1. 明确 face_db.json 作为唯一业务真相源。
2. descriptors.yml 仅作为性能缓存，不承载独立语义。
3. 启动时增加数据库与缓存的一致性检查和自动修复。

### 3.2 readiness 语义仍偏弱

status.vision.ready 使用 PUB/SUB 单向广播，不具备可靠送达保证。虽然本轮已经通过“重复 ready 广播 + 状态事件兜底”显著降低了偶发失败，但如果后续 Daemon 需要更强的就绪确认，仍建议增加同步 health query，例如:

- query.vision.health
- query.vision.runtime_state

## 4. 剩余风险

### 4.1 合成视频上的冷启动已注册识别不稳定

当前测试方案将“冷启动已注册 known_subject 在合成视频上立刻命中”保留为观测项，而不是阻断验收的强校验项。

原因:

- 合成视频会引入缩放、压缩、插值和运动轨迹变化
- 初始样本仅一张，覆盖面有限
- 这条链路不是“注册后热更新识别”主能力的必要前提

建议:

1. 为真实摄像头输入补充单独场景测试。
2. 为已注册身份准备多样本数据集，而不是单样本。
3. 在日志中补充 top-1 score 和 threshold，便于后续调阈。

### 4.2 识别过程日志噪声较大

当前 ncnn 线程数相关日志较多，会淹没关键控制日志。建议后续做两件事:

1. 区分 debug 日志和验收日志级别。
2. 在测试模式下提供更干净的摘要输出开关。

---

## 5. 建议的后续改进优先级

### P1

- 增加 query.vision.health，提供同步就绪检查
- 为识别输出 top-1 score、top-2 score、threshold 诊断信息
- 统一 face_db 与 descriptor cache 的事实来源关系

### P2

- 增加 recapture 的自动化功能测试
- 增加 query.vision.face.position 的自动化功能测试
- 增加多样本已注册身份数据集，提升冷启动识别稳定性

### P3

- 清理 deprecated zmq setsockopt API 调用
- 收敛调试日志输出，降低大体量测试日志噪声

---

## 6. 结论

本轮工作的核心结论如下:

1. FaceReco 之前的主要问题不在“有没有接口”，而在“接口是否真正闭环、是否可自动验证”。
2. 通过修复运行时注册、EOF 状态、NaN 描述子和 ready 稳健性后，模块已经具备可执行、可观察、可回归的功能测试主路径。
3. 当前最重要的剩余工程问题，不是基础功能缺失，而是数据源统一、health 语义增强和真实场景识别稳定性提升。

在当前版本下，作为 ZMQ 控制的人脸视觉服务，FaceReco 已达到可以继续向上层 Daemon 集成和持续回归的状态。
